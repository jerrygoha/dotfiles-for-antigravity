# /brainstorm 완벽 상세 가이드

> 여러 대안을 탐색하고 최적의 방안을 선택하는 워크플로우

---

## 📌 개요 및 목적

`/brainstorm`은 **여러 가지 접근법을 비교**하고 최적의 방안을 선택하는 워크플로우입니다.

**언제 결정이 어려운가요?**
- 기술 선택 (라이브러리 A vs B)
- 아키텍처 결정 (모노리스 vs 마이크로서비스)
- 구현 방식 (여러 방법이 있을 때)

---

## 🎯 언제 사용하나요?

### ✅ 이럴 때 사용하세요

| 상황 | 예시 |
|------|------|
| 기술 선택 | "상태관리 Redux vs Zustand?" |
| 아키텍처 결정 | "API 설계를 어떻게?" |
| 방법이 여러 개 | "이 문제 어떻게 풀지?" |
| 트레이드오프 분석 | "성능 vs 개발 속도" |

### 📋 구체적인 사용 시나리오

```
"실시간 알림 기능을 추가하려는데,
WebSocket, SSE, Polling 중 뭐가 좋을까요?"

→ /brainstorm 사용
→ Phase 1: 문제 명확화
→ Phase 2: 3가지 옵션 분석
→ Phase 3: 가중치 평가
→ Phase 4: 추천
```

## 📤 기대 결과물

```markdown
## Problem
실시간 알림 시스템: 서버→클라이언트 푸시 필요

## Options

### Option 1: WebSocket
- **Approach**: 양방향 연결 유지
- **Pros**: 실시간, 양방향
- **Cons**: 서버 리소스, 연결 관리 복잡
- **Effort**: High
- **Score**: 4.2

### Option 2: Server-Sent Events (SSE)
- **Approach**: 단방향 스트리밍
- **Pros**: 간단, 자동 재연결
- **Cons**: 단방향만, IE 미지원
- **Effort**: Low
- **Score**: 4.5

### Option 3: Long Polling
- **Approach**: 주기적 요청
- **Pros**: 단순, 호환성
- **Cons**: 지연, 서버 부하
- **Effort**: Low
- **Score**: 3.2

## Evaluation Matrix

| Criteria | Weight | WS | SSE | Polling |
|----------|--------|----|----|---------|
| 실시간성 | 0.3 | 5 | 4 | 2 |
| 구현 복잡도 | 0.25 | 2 | 4 | 5 |
| 서버 부하 | 0.25 | 3 | 4 | 2 |
| 확장성 | 0.2 | 4 | 4 | 3 |

## Recommendation
**SSE (Score: 4.5)** 권장

이유: 단방향 알림에 충분하고 구현이 간단함.
양방향 필요시 추후 WebSocket으로 마이그레이션.
```

## 📖 상세 사용법

### Step 1: 문제 제시

```
/brainstorm 인증 방식을 결정해야 해.
JWT vs Session 중 뭐가 좋을까?
우리는 모바일 앱도 지원해야 해.
```

### Step 2: 명확화 질문 응답

에이전트가 질문할 수 있습니다:
- "예상 동시 사용자 수는?"
- "보안 민감도는?"
- "기존 인프라는?"

### Step 3: 옵션 검토

각 옵션의 Pros/Cons/Effort 확인

### Step 4: 가중치 조정 (선택)

```
"보안을 더 중시해줘. 가중치 0.4로"
```

### Step 5: 결정

```
"좋아, Option 2로 가자"
→ /write-plan으로 이어가기
```

## ⚠️ 이런 상황엔 다른 워크플로우를!

| 상황 | 대신 사용할 워크플로우 |
|------|----------------------|
| 결정 완료, 계획 필요 | `/write-plan` |
| 기술 조사 필요 | `/research` |
| 구현만 하면 됨 | `/tdd` 또는 `/execute-plan` |

### 혼동하기 쉬운 상황

**Q: `/brainstorm` vs `/research`?**
- `/brainstorm`: 옵션 비교 및 결정
- `/research`: 정보 수집 및 문서화

**Q: `/brainstorm` vs `/write-plan`?**
- `/brainstorm`: "**뭘** 할지" 결정
- `/write-plan`: "**어떻게** 할지" 계획

## 🔗 함께 사용하면 좋은 워크플로우

### 의사결정 흐름

```
/research            # 기술 조사 (필요시)
    ↓
/brainstorm          # 옵션 비교 및 결정
    ↓
/write-plan          # 구현 계획
    ↓
/execute-plan        # 실행
```

## 💡 프로 팁

1. **최소 3개 옵션**: 2개는 이분법적, 3개 이상이 좋음
2. **가중치 커스텀**: 프로젝트에 맞게 조정
3. **반대 의견 고려**: Cons도 충분히 검토
4. **결정 기록**: ADR(Architecture Decision Record)로 문서화
