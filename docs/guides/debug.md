# /debug 가이드

> TDD 기반 체계적 디버깅으로 근본 원인을 찾는 워크플로우

---

## 📌 핵심 원칙

> ❌ "이게 원인인 것 같다" → 수정  
> ✅ "테스트로 재현 → 원인 확인 → 수정 → 테스트 통과"

---

## 📌 언제 사용하나요?

| 상황 | 예시 |
|------|------|
| 버그 발생 | "로그인할 때 에러 나요" |
| 원인 불명확 | "가끔 데이터가 안 나와요" |
| 간헐적 실패 | "결제가 가끔 안 돼요" |
| 복잡한 로직 오류 | "계산 결과가 이상해요" |

---

## 📤 6단계 프로세스

### Phase 0: 재현 테스트 (필수!)
**수정 전에 실패하는 테스트를 먼저 작성**

### Phase 1: 증거 수집
에러 로그, 스택 트레이스, 입력 데이터 수집

### Phase 2: 가설 형성 (최소 3개)
증거 기반으로 가설 수립, 추론만으로 X

### Phase 3: 테스트로 가설 검증
각 가설을 테스트 코드로 검증

### Phase 4: 수정
확인된 원인에 대해서**만** 수정

### Phase 5: 검증 & Regression
재현 테스트 PASS + 전체 테스트 PASS 확인

### Phase 6: 문서화
Bug Resolution Report 작성

---

## ⚠️ 안티패턴

| ❌ 하지 말 것 | ✅ 해야 할 것 |
|-------------|-------------|
| 추론만으로 수정 | 테스트로 원인 확인 후 수정 |
| 여러 가설 동시 수정 | 한 번에 하나씩 검증 |
| 수동 확인만 | 자동화 테스트로 검증 |
