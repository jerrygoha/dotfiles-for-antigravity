# /research 가이드

> 다중 출처 검증 기반의 기술 조사 워크플로우

---

## 📌 핵심 원칙

> **단일 출처 의존 금지.** 모든 주장은 최소 2개 이상 독립 출처로 검증.

---

## 📌 언제 사용하나요?

| 상황 | 예시 |
|------|------|
| 새 기술 평가 | "Next.js 15 새 기능 조사" |
| 라이브러리 비교 | "상태관리 라이브러리 비교" |
| 아키텍처 결정 근거 | "서버리스 vs 컨테이너" |
| 복잡한 버그 조사 | "이 에러 해결법 찾아줘" |

---

## 📤 출처 신뢰도 계층

| Tier | 출처 유형 | 신뢰도 | 예시 |
|------|---------|--------|-----|
| 1 | 공식 문서 | ⭐⭐⭐⭐⭐ | MDN, RFC, 공식 Docs |
| 2 | 1차 출처 | ⭐⭐⭐⭐ | GitHub repo, 저자 블로그 |
| 3 | 큐레이션 | ⭐⭐⭐ | Awesome 리스트, 기술 블로그 |
| 4 | 커뮤니티 | ⭐⭐ | Stack Overflow, 포럼 |
| 5 | AI/집계 | ⭐ | **반드시 상위 Tier로 검증** |

---

## 📤 검증 템플릿

```markdown
## Claim: "[주장 내용]"

| 출처 | 유형 | URL | 확인됨? |
|------|------|-----|---------|
| [이름] | Tier 1 | [URL] | ✅/❌ |
| [이름] | Tier 2 | [URL] | ✅/❌ |

**Verdict**: Verified / Partially / Unverified
```

---

## ⚠️ 안티패턴

| ❌ 하지 말 것 | ✅ 해야 할 것 |
|-------------|-------------|
| 단일 출처만 인용 | 최소 2개 독립 출처 |
| "~라고 알려져 있다" | "[출처]에 따르면 ~" |
| 장점만 나열 | 단점/리스크도 조사 |
| AI 출력 그대로 | Tier 1-3으로 검증 |
